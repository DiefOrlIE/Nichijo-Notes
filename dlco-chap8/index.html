背景：CPU 在干什么？冯·诺依曼模型的极简版

补充背景 · 进入第一讲之前

你已经写过 C 代码了。但代码到底是怎么"跑起来"的？这一章的所有图，本质都在回答一个问题：从一条指令变成电信号在线路上跑，中间发生了什么？

三个角色，一辈子只做三件事

  1. 存储器 Memory：一个大数组，地址 → 数据。存指令和数据。
  2. CPU：一个工人。手里只能干"读/写存储器""做算术""跳到别处"三件事。
  3. I/O：和外部世界交换数据。本章基本不涉及。

"取指-译码-执行"循环

CPU 永远在跑一个 while(true)：

while (true) {
    指令 = Memory[PC];   // 取指
    PC = PC + 4;          // 顺序往后
    译码(指令);            // 看懂这是什么操作
    执行(指令);            // 操作 ALU、读写寄存器、读写内存
    if (异常) 跳转到异常处理;
    if (中断请求) 跳转到中断处理;
}
关键直觉：PC（程序计数器）是 CPU 的"手指头"，指向下一条要执行的指令。改变 PC 就改变了程序走向（这就是 jump/branch 的本质）。

本章在解决什么

给你一份 ISA（指令集，比如 RISC-V），让你用数字电路真的造一个能跑这个循环的硬件。两个主要部分：

  • 数据通路 Datapath：搬运、计算数据的"管道+部件"（ALU、寄存器、MUX 等）
  • 控制器 Control Unit：根据当前指令，决定每个部件该不该工作、怎么工作（发出"控制信号"）
第一讲只讲两类"元件"和"时序约束"——为后面真正动手设计数据通路打地基。

CPU 执行程序和指令的过程

PDF 第 3 页

把上面的 while 循环展开成 10 个小步骤

每条指令的生命周期分成两大阶段：

  • 取指阶段：① 取指令 (从内存 M[PC] 取出来放进 IR) ② PC+1 送 PC（这里"1"是抽象的，对 RV32I 实际是 PC+4，因为指令 32 位 = 4 字节）
  • 译码 & 执行阶段：③ 译码 → ④ 算地址 → ⑤ 取操作数 → ⑥ ALU 运算 → ⑦ 存结果 → ⑧ 检测异常 → ⑨ 异常处理 → ⑩ 检查中断
异常 vs 中断：异常由 CPU 内部产生（除零、非法指令、溢出）；中断由外部硬件触发（键盘、计时器、I/O 完成）。这俩处理机制类似，但触发源完全不同。

为啥要分这么细

因为每一步在硬件上对应一个或多个"动作"。后面要做单周期、多周期、流水线，本质都是"这些步骤怎么排进时钟周期"的问题。记住这 10 步，是理解后面所有数据通路图的基本盘。

CPU 基本结构 · Control Unit vs Datapath

PDF 第 5 页

CPU 的两大分工

把上一页的"全家福"打包，CPU 内部其实只有两块：

  • Datapath（数据通路）：所有能搬数据/算数据的部件。它是指令的执行部件——干活的。
  • Control Unit（控制器）：根据指令译码结果，决定 Datapath 中每个 MUX 选哪个输入、ALU 做什么操作、要不要写回寄存器……它是指令的控制部件——指挥的。
类比：Datapath 是一条工厂流水线（带传送带、机床、机械手），Control Unit 是厂长——根据生产订单（指令）给每台机器下指令"现在干这个，下一秒换那个"。

为啥要做这个划分

因为设计方法完全不一样：

  • Datapath = 元件 + 连线 → 组合 + 时序电路问题
  • Control Unit = 输入是指令，输出是一堆 0/1 控制信号 → 布尔函数问题（真值表 → 逻辑表达式 → 电路）

所以接下来：第二讲专门搞 Datapath，第三讲专门搞 Control Unit。这套切分贯穿整章。

背景：组合电路 vs 时序电路（你最需要的复习）

补充背景 · 进入 p6 之前必看

下面整章都会用"操作元件 = 组合"、"状态元件 = 时序"两个词。先把这俩搞清楚，后面才不会懵。

组合逻辑（Combinational Logic）

  • 没有记忆。输出 = 输入的纯函数（boolean function）。同样的输入永远给同样的输出。
  • 典型：与门、或门、非门、加法器、MUX、Decoder、ALU。
  • 不需要时钟！信号一进来，经过几纳秒的"门延迟"，输出就稳定了。
输出(t) = f(输入(t))

时序逻辑（Sequential Logic）

  • 有记忆。输出不只看现在的输入，还看"上次存了什么"。
  • 典型：D 触发器、寄存器、寄存器组、内存。
  • 必须有时钟！时钟边沿到来时，才把"输入"写进去变成"新状态"。
下一状态(t+1) = f(当前输入(t), 当前状态(t))

为什么数字电路要这么分

因为"计算"和"记住"是两件不同的事。组合电路负责计算（瞬时的），时序电路负责记忆（跨时钟周期的）。一个 CPU = 一堆组合电路 + 一堆时序电路 + 时钟 + 控制信号。

本章的命名：
• 操作元件 = 组合元件 = 算东西的 → ALU/Adder/MUX/Decoder/扩展器
• 状态元件 = 时序元件 = 存东西的 → PC/IR/MAR/MDR/GPRs/Memory
易错：寄存器组的"读"是组合操作（地址进去，数据立刻出来，不要时钟）；"写"才是时序操作（要时钟边沿）。这点在 p10 会反复出现。

数据通路的基本结构

一句话总结

数据通路 = 操作元件（组合电路）+ 存储元件（时序电路）+ 连线方式（总线 or 分散）。功能就是"取数 → 数据运算 → 存数"循环。

这页是承上启下的索引页。接下来 p7-p12 会一个一个细讲这些元件。

操作元件：组合逻辑电路

PDF 第 7 页

四个最常见的组合元件

  • Adder（加法器）：输入 A、B、Cin（进位输入），输出 Sum、Carry。一次只能干"加法"。
  • MUX（多路选择器）：N 个输入选 1 个输出，由 Select 控制信号决定选谁。MUX 是数据通路的"红绿灯"，遇到岔路就放它。
  • ALU（算逻部件）：万能加法器升级版，由 OP 控制信号决定做哪种运算（加/减/与/或/比较……）。还会顺带产生 Zero 信号告诉外面"结果是不是 0"。
  • Decoder（译码器）：N 位输入 → 2^N 位输出，选中其中一根线变 1（用来"选中某个寄存器/某个状态"）。

组合元件的两条特征（必背）

① 输出只取决于当前输入：输入一样，输出一样（无记忆）。
② 定时：输入到达后，经过一定的"门延迟"输出就改变，不需要时钟。
图上虚线红线 = 控制信号。Select、OP 都是控制信号，由控制器从指令译码后产生（不是从指令字段直接给的）。

状态元件：时序逻辑电路

PDF 第 8 页

"状态元件"是啥

能存住一个值，跨过时钟周期仍然保留——这就是"状态"。一旦能存值，就需要规定"什么时候允许它改变"，否则它会乱跳，整个 CPU 失控。

边沿触发（edge-triggered）

  • 规定：状态元件中的值只在时钟边沿改变，每个时钟周期最多变一次。
  • 上升沿触发：时钟从 0 跳到 1 的瞬间读/写。
  • 下降沿触发：时钟从 1 跳到 0 的瞬间读/写。
为什么用"边沿"而不是"高电平"：边沿是一个瞬间，能精确定义"何时发生"。如果用"高电平期间随便写"，那一个时钟周期内可能写好多次，无法确定结果。

波形图上 cycle time = 一个完整周期的长度。整个 CPU 的"心跳"就是这个时钟。

最简状态元件 = D 触发器：一个时钟输入、一个数据输入 D、一个状态输出 Q。下一页详讲。

存储元件何时被改变？Setup / Hold / Clk-to-Q

PDF 第 9 页 · 重点

D 触发器的三个时间参数（一定要记死）

参数含义类比
Setup Time时钟边沿到来之前，输入 D 必须稳定多久考试前要提前到考场（提早稳定）
Hold Time时钟边沿到来之后，输入 D 必须再保持稳定多久考完不能立刻走，要等监考完成确认
Clk-to-Q时钟边沿到了，Q（输出）不会立刻变，要等这点延迟才能反映新值从下单到上菜的延迟
切记：状态元件的输入信息总是在一个时钟边沿到达后的 Clk-to-Q 时刻才被写入，此时输出才反映新的状态值。这意味着——你在时钟边沿"前一瞬间"读到的 Q 还是旧值！

为什么要懂这个

后面（p16-18）算"时钟周期最少要多长"就靠这三个参数。如果时钟周期太短，没等数据稳定就被锁住，结果就是错的。所以芯片主频不能无限提高——物理延迟是底线。

数据通路里的状态元件有两种

① 单个寄存器（PC、IR、MAR、MDR）
② 寄存器组 / 存储器（GPRs、Memory）
下一页分别讲。

存储元件：寄存器 & 寄存器组（Register File）

PDF 第 10 页

单个寄存器 Register

  • 有一个 Write Enable (WE) 信号控制要不要写：
  • WE=0：时钟边沿来时输出不变（这次时钟"无视"输入）
  • WE=1：时钟边沿来时开始把输入写入
  • 如果每个时钟边沿都要写（如 PC），就不需要 WE 信号了

寄存器组 Register File（GPRs 的实现）

RV32I 有 32 个 32 位通用寄存器。需要：

  • 两个读口（busA、busB）：因为一条指令最多读 2 个源操作数（rs1, rs2）。读口由 RA、RB 给出地址（各 5 位，2⁵=32）。读是组合操作，地址给定后经一个 "access time" 数据就出来了，不用等时钟。
  • 一个写口（busW）：写回结果 R[rd]。写是时序操作，需要 WE=1 且时钟边沿到来时才生效。
为啥读组合而写时序：因为读不会改变状态，所以可以随时读（地址变，数据立刻跟着变）。写会改变状态，必须严格规定"什么时候改"，否则同一周期里值可能被改好几次，乱套。
"取数时间 (Access Time)"≈ 寄存器组内部 Decoder + MUX 的延迟。这是计算关键路径的一部分（后面 p81 会用到）。

寄存器组的内部结构

PDF 第 11 页

把图剖开看

  1. 左侧 32-to-1 Decoder：RW (5 位) 进来 → 解码出 32 根线 → 选中第 RW 个寄存器，让它准备接收数据。
  2. 每个 Register 旁的 AND 门：把 WE 信号和 Decoder 选中信号与起来——只有"WE=1 且 被选中"才会触发写。
  3. busW：要写入的数据，并行送到所有寄存器的 D 输入端（具体只有被选中的那个会真写进去）。
  4. 右侧两个 32-to-1 MUX：分别由 RA、RB 控制——选择哪个寄存器的输出送到 busA、busB。这就是"两个读口"的硬件实现。
为什么需要两个读 MUX？因为想同时读两个寄存器（如 add 需要读 rs1 和 rs2）。两个 MUX = 两个独立的读通道。

整张图说明：寄存器组 = 32 个寄存器 + 1 个写 Decoder + 2 个读 MUX + 一堆 AND。它本身就是一个"组合 + 时序"的混合体。

同步系统（Synchronous System）

所有动作有专门时序信号来定时——告诉每个部件"现在该做什么、做多久"。最常用的时序信号就是时钟。

四个层次的"时间单位"（早期计算机概念）

  • 指令周期：取并执行一条指令所花总时间。不同指令的指令周期可能不同。
  • 机器周期：一个指令周期分成的几个大阶段（取指、读操作数、执行写回、中断响应……）
  • 节拍 (P0/P1/P2/P3)：机器周期内的"小阶段"（下一页波形）
  • 脉冲 (C0/C1/C2/C3)：节拍内更细的定时信号
这是早期三级时序系统。现代 CPU 不再用机器周期和节拍这么细的层次——直接用单一时钟，"一个时钟周期就是一个节拍"。所以这页主要是历史知识，了解即可。

现代计算机的时钟周期：Cycle Time 公式

PDF 第 16 页 · 重点

"时钟周期到底要多长"这个核心问题的答案

三明治结构：状态元件 → 组合逻辑 → 状态元件。下一个时钟边沿到来前，数据必须从左边触发器跑过组合逻辑，到达右边触发器并满足 Setup。

Cycle Time = Latch Prop (Clk-to-Q) + Longest Delay Path + Setup + Clock Skew

四项各是啥

  • Latch Prop / Clk-to-Q：源触发器从时钟边沿到输出稳定的延迟
  • Longest Delay Path：组合逻辑的最长路径延迟（最坏情况）
  • Setup：目的触发器要求的"提前稳定时间"
  • Clock Skew：时钟到不同触发器的时间差（物理上时钟线不一样长）
记忆要点：时钟周期 = 起跑延迟 + 跑步距离 + 终点要求稳定的时间 + 跑道偏差。任意一项变大，主频就得降。这就是为什么芯片提频这么难。

另一条约束（防止跑得太快）

Latch Prop + Shortest Delay Path − Clock Skew > Hold Time

意思：如果组合逻辑跑得太快，新数据可能"赶在前一个周期的 Hold Time 结束之前"就到了目的触发器，破坏了它的保持要求。

回顾：电路设计与分析（时序波形）

四条信号轨道的故事

  1. CLOCK：方波
  2. 触发器输出：时钟边沿后经 t_ffpd 才稳定（叉号区域是"还在变"）
  3. 组合电路输出：触发器输出经过组合电路又延迟 t_comb，才稳定
  4. 触发器输入：上一步的稳定值正是下一级触发器的输入

两个"容限"是什么

  • 建立时间容限 = t_clk − t_ffpd(max) − t_comb(max) − t_setup。剩余的"余地"。>0 才合法。
  • 保持时间容限 = t_ffpd(min) + t_comb(min) − t_hold。也要 >0。
"max" vs "min"：max 用在建立约束（最坏情况下数据要赶到），min 用在保持约束（最好情况下数据也不能跑太快冲掉旧值）。

两条时序约束（铁律）

PDF 第 18 页
(1) t_clk > t_ffpd(max) + t_comb(max) + t_setup
(2) t_hold < t_ffpd(min) + t_comb(min)

口语版

  • 约束 (1) "别太慢"：时钟周期得够长，让数据走完最慢路径还能满足 Setup。
  • 约束 (2) "别太快"：组合逻辑也不能跑太快，否则新数据冲掉了旧数据还没保持够 Hold。
两条约束都得满足。约束(1)决定最高主频；约束(2)决定能不能用某些"快得过头"的逻辑（比如非常浅的组合电路加在两个触发器之间，得故意延迟一下）。

CPU 执行时间的计算（核心公式）

PDF 第 21 页 · 必背

三大公式拆解

CPU 时间 = 时钟周期数 / 程序 × 时钟周期
= 时钟周期数 / 程序 ÷ 时钟频率
= 指令条数 × CPI × 时钟周期

三个因子各是啥

  • 指令条数 IC (Instruction Count)：程序总共要执行多少条指令。由编译器+ISA共同决定。
  • CPI (Cycles Per Instruction)：平均每条指令花几个时钟。由组织结构（流水线深度、多发射）+编译器（指令选择）决定。
  • 时钟周期 t_clk = 1/频率：一个周期多长。由组织+工艺技术决定。
这是体系结构的"圣三位一体"：要降低 CPU 时间，要么减少指令条数（更智能的 ISA / 编译器），要么降 CPI（更好的微架构），要么缩短时钟周期（更先进的工艺）。三者通常互相牵制——提高主频常常导致 CPI 也上升。

CPI 衡量啥

CPI 反映 ISA + Organization + Compiler/Algorithm 的综合结果。

如何计算 CPI（程序综合 CPI）

PDF 第 25 页

三种 CPI 的层次

  • 某条特定指令的 CPI：一个固定值（设计时确定）
  • 某个程序的 CPI：该程序所有指令的加权平均
  • 某机器的 CPI：某种典型负载下的平均
总时钟数 = Σ (CPIᵢ × Cᵢ)
程序 CPI = Σ (CPIᵢ × Fᵢ)，其中 Fᵢ = Cᵢ / 总指令数

已知 CPU 时间、时钟频率、指令数，反推 CPI：
CPI = (CPU时间 × 时钟频率) / 指令条数

常见考点：给你"22% Load、11% Store、49% R-Type……"和各自 CPI，让你算综合 CPI。就是加权平均。p100 会做这个题。
单靠 CPI 看不出性能！ 单周期处理器 CPI=1，但时钟周期超长，性能反而差。性能 = IC × CPI × t_clk，三者都得看。

Example 1：频率 vs CPI 的权衡

PDF 第 26 页 · 必练例题

题目

程序 P 在机器 A 上跑 10s，A 的频率 400MHz。机器 B 想让 P 只跑 6s，但 B 的高频会让 CPI 变大 → 程序 P 在 B 上的时钟周期数是 A 上的 1.2 倍。问 B 的频率要达到 A 的几倍？

解题步骤（务必跟着算一遍）

  1. A 的时钟周期数 = CPU时间 × 频率 = 10s × 400MHz = 4000M 个
  2. B 的时钟周期数 = 1.2 × 4000M = 4800M 个
  3. B 的频率 = 周期数 / CPU时间 = 4800M / 6s = 800 MHz
  4. 所以 B 频率是 A 的 2 倍
反直觉的结论：B 频率虽是 A 的 2 倍，但性能只快了 10/6 = 1.67 倍——因为 CPI 涨了 1.2 倍把好处吃了一部分。这就是"提频不是免费午餐"的数值演示。

Marketing Metrics：MIPS 和 MFLOPS 的局限

MIPS：每秒百万条指令

MIPS = IC/秒 × 10⁻⁶ = ClockRate / CPI × 10⁻⁶

厂商喜欢拿来宣传，但有问题：

  • 不同机器指令集不同，一条 ARM 指令 ≠ 一条 x86 指令。比较 MIPS 不公平。

  • Peak MIPS（峰值）几乎不实用。

专门衡量浮点性能，但不一定反映程序整体性能（如果你的程序大部分是整数操作，MFLOPS 高也没用）。

真正可比的指标是跑同一个 benchmark 程序所需的时间（如 SPEC），而不是 MIPS/MFLOPS。

扩展器：5 种立即数拼接方式

PDF 第 34 页

为啥要扩展器

RV32I 指令是 32 位，但立即数字段最多 20 位（一般 12 位）。要参与 32 位 ALU 运算，必须先扩展到 32 位。而且不同指令格式的立即数位置和拼接方式都不同，所以需要一个统一的"立即数拼接器"。

5 种拼接方式（Verilog 语法 {...} 表示位拼接）

  • immI = {20{Instr[31]}, Instr[31:20]} — 取 [31:20]，高位用符号位 Instr[31] 复制 20 次填充
  • immU = {Instr[31:12], 12'b0} — 高 20 位放原位，低 12 位补 0（lui 专用）
  • immS = {20{Instr[31]}, Instr[31:25], Instr[11:7]} — Store 立即数被拆在两段
  • immB = {20{Instr[31]}, Instr[7], Instr[30:25], Instr[11:8], 1'b0} — Branch 立即数四段拆，且最低位强制 0（2字节对齐）
  • immJ = {12{Instr[31]}, Instr[19:12], Instr[20], Instr[30:21], 1'b0} — Jump 立即数同理
为啥 B/J 立即数最低位强制 0：分支/跳转目标必须是偶数地址（指令长度 2 字节或 4 字节对齐），所以最低位必为 0，干脆从编码中省掉了，硬件再补回去。这多出 1 位的跳转范围。

右上 MUX：5 个输入 (immI/U/S/B/J) 选一个输出，由控制信号 ExtOp（3 位）选择。

背景：什么是"控制信号"？它来自哪里？

补充背景 · 进入 p36 之前

控制信号的本质

控制信号 = 由控制器输出、用来"使能/选择"数据通路中各部件行为的 0/1 信号。

例子

  • RegWr（1 位）：要不要写回寄存器组。0=不写，1=写
  • ALUctr（4 位）：让 ALU 做什么操作（加/减/与/或/比较）
  • ALUBSrc（2 位）：ALU 的 B 输入端 MUX 选哪个（寄存器/立即数/"4"）
  • MemWr：要不要写数据存储器
  • MemtoReg：写回寄存器的值是来自 ALU 还是 内存
  • Branch/Jump：是否要改 PC

来源

控制信号不是从指令字段直接取出来的。控制器拿到 opcode（和有时 funct3/funct7），译码后生成这些信号。本质上：

控制信号 = f(opcode, funct3, funct7)

f 是一组布尔函数，可以用真值表表示，再用PLA（可编程逻辑阵列）实现。第三讲（p73-79）详讲这步。

记忆法：图上的虚线红线

第二讲数据通路图里，所有虚线红色信号都是控制信号。它们从图外的"控制器盒子"伸进来——你画数据通路时只需要标"这里需要一根 ALUctr"，不用管它从哪生成。生成是控制器的活。

分工心法：数据通路负责"如果给我这些控制信号，我就能正确执行任何指令"；控制器负责"我能根据 opcode 生成正确的控制信号"。各管一段，互相解耦。

算术逻辑部件 ALU 的设计

PDF 第 36 页 · 关键图

这个 ALU 能做什么

add, or, slt, sltu, srcB（直通 B）, 判 0（sub）——这 6 种操作覆盖了 9 条指令的所有计算需求。

9 条指令分别用哪种 ALU 操作

  • add/lw/sw/jal: add（lw/sw 算地址也是加）
  • ori: or
  • slt: slt; sltu: sltu
  • lui: srcB（直接把扩展后的立即数透传到输出）
  • beq: 判 0（即 sub 然后看 Zero 标志）

图上三个子控制信号

  • SUBctr：是否减法（slt/sltu/sub 时为 1）
  • SIGctr：是否带符号比较（slt=1, sltu=0）
  • OPctr：选择最后输出 MUX（加=00, 或=01, srcB=10, slt/sltu=11）

这三个子信号都由 ALUctr (4 位) 通过 "ALU 操作控制信号生成部件"（一个组合逻辑小盒子）派生出来。

设计直觉：ALU 本身只是个组合电路集合体。通过给加法器加一些前置 XOR/OR 门，"加法器"可以同时实现加、减、比较。这是数字电路课的经典技巧（减法 = 取反加 1）。

R-型指令的数据通路

R 型功能

R[rd] ← R[rs1] op R[rs2]。三条 R 型：add/slt/sltu，分别对应 ALU 三种操作。

数据流

  1. PC 取指 → 拿到 32 位指令 Instr
  2. Instr[24:20]=rs2 送 Rb，Instr[19:15]=rs1 送 Ra，Instr[11:7]=rd 送 Rw（写回地址）
  3. 寄存器组读出 busA=R[rs1], busB=R[rs2]
  4. 送入 ALU，ALUctr 由 opcode + funct3 决定

三条 R 型都要写结果，所以 RegWr=1。ALUctr: add=0000 / slt=0010 / sltu=0011。

怎么判相等

下地址逻辑的具体实现

PDF 第 52 页

下地址逻辑盒子里有什么

看上半放大图：

  • 一个 MUX：选"4"或"imm"作为加数
  • 一个 Adder：PC + 选中的加数 → 下条 PC 候选
  • 一个 AND 门：Branch & Zero → 控制 MUX

逻辑：

  • Branch=0：MUX 选"4"，PC←PC+4（普通顺序）
  • Branch=1 & Zero=1：MUX 选"imm"，PC←PC+imm（跳转成立）
  • Branch=1 & Zero=0：MUX 选"4"，PC←PC+4（跳转不成立）
这就是分支的硬件实现。一个 MUX + 一个 AND 门，搞定。

一个细节：旧控制信号会不会捣乱？

取指时，新指令还没译码，控制信号还是上条指令的旧值。会不会出问题？

答案：不会。因为在下个 Clk 到来之前，PC 输入端的值不会真的写入。只要保证下个 Clk 来临前能产生正确的 PC 值即可。也就是说——译码、控制信号生成、下址逻辑计算都在这个时钟周期内并行进行，到周期末才一起锁存。

图上 Jump=previous, Branch=previous, Zero=previous 都是"上条指令留下来的"，但 MUX 选择的下条 PC 在整个周期结束前才被使用，所以无害。

  • ALUASrc=×（A 端不重要）, ALUBSrc=2, ALUctr=srcB, ExtOp=immU

lw 执行过程

lw 的完整路径

取指 → 读 rs1 → busA → ALU A 口；扩展器 (immI) → ALU B 口 → ALU add → Addr 送数据存储器 → 读出数据 → MUX (MemtoReg=1 选内存) → busW → 写回 rd。

sw 执行过程

取指 → 读 rs1（地址基址）+ rs2（要写的数据）→ ALU 算地址（rs1+immS）→ 写入数据存储器（MemWr=1, Data In = busB = R[rs2]）。

  • ALUASrc=0, ALUBSrc=2, ALUctr=add, ExtOp=immS
  • MemWr=1, MemtoReg=×, RegWr=0（不写寄存器）

beq 执行过程

jal 执行过程 · 跳过细讲

jal 的执行过程已在 p53 详讲。这页核心要点：

  • ALUASrc=1（A 选 PC，因为要算 PC+4 写回 rd）
  • ALUBSrc=1（B 选"4"）
  • ALUctr=add → 结果 = PC+4 → 写回 rd（保存返回地址）

背景：真值表 → 逻辑表达式 → PLA 的链路

补充背景 · 进入 p73 之前

布尔逻辑设计的标准三步

  1. 列真值表：把所有"输入组合 → 输出"列出来
  2. 化简成逻辑表达式：积之和 (SOP) 或卡诺图化简
  3. 用 AND/OR/NOT 门搭出电路，或者用 PLA（可编程逻辑阵列）批量实现

PLA 是啥

PLA = Programmable Logic Array。结构：

  • 上半：AND 平面。每根竖线代表一个"乘积项"（minterm），通过点连接选择哪些输入参与
  • 下半：OR 平面。每根横线代表一个输出，通过点连接哪些乘积项要"或"起来

优点：把所有控制信号的逻辑表达式放在同一个矩形电路里，整齐、易修改。

为啥用 PLA 而不是离散门电路：当输出多、输入多时，PLA 排布规整、面积可控、可重新编程。CPU 控制器有大量类似结构的布尔函数，PLA 完美适配。

SOP 形式回顾

任意布尔函数都可以写成"乘积项之和"。例如：

F = A·B·C  +  ~A·B·~C  +  A·~B·C

每个项是若干输入的 AND，所有项 OR 起来。这恰好对应 PLA 的 AND 平面 + OR 平面。

时钟周期的确定：关键路径分析

关键路径 (Critical Path)

所有指令中路径最长的那条决定时钟周期。对单周期 RV32I 子集，这条路径是 Load 指令：

Critical Path (Load) =
PC's prop time (Clk-to-Q) + Instruction Memory's Access Time + Register File's Access Time + ALU to Perform a 32-bit Add + Data Memory Access Time + Setup Time for Register File Write + Clock Skew

6 段延迟串行累加

  1. PC 输出稳定（Clk-to-Q）
  2. 取指（IMem Access）
  3. 读寄存器（RegFile Access）
  4. ALU 算地址（ALU Add）
  5. 读数据存储器（DMem Access）
  6. 满足寄存器组 Setup 写回

+ 时钟偏移容限 Clock Skew。

每一段都不能省——单周期就是这么憋屈：哪怕只是 add 指令，也得等够这么长的时钟周期才能进下一拍。

多周期处理器的实现思想

PDF 第 87 页

单周期的痛点

时钟周期 = 最复杂指令的执行时间 → 太长。短指令也得等。

多周期的思路

  1. 把指令分成若干阶段，每阶段在一个时钟周期内完成
  2. 时钟周期 = 最复杂阶段所花时间（远小于最长指令的总时间）
  3. 尽量让各阶段时长均衡
  4. 每阶段只做一种事：访存 OR 寄存器读/写 OR ALU 运算（其中一项）

多周期的好处

  • 时钟周期短（虽然 CPI 变大）
  • 不同指令周期数不同：Load 用 5 拍，Jump 用 3 拍（短指令真正快了）
  • 功能部件可复用：只需要 1 个 ALU，不同周期用同一个 ALU 做不同事（取指算 PC+4 也用它）；IMem 和 DMem 可合并为一个 Memory
多周期的核心思想：用更细的时钟切割，让每个时钟周期都"干刚好够用的一件事"。短指令少用几拍，长指令多用几拍。

每步执行结果需要存到临时寄存器（下个时钟周期才能用）。这是多周期相比单周期的额外开销。

关键区别

  • 硬连线：直接用组合逻辑电路实现 FSM。速度快，但复杂指令集时电路庞杂。
  • 微程序：把每个状态的控制信号编码成微指令，存在 ROM 里，像跑程序一样按 μPC 顺序读出。规整、易修改，但慢。

硬连线 vs 微程序的特点

PDF 第 91 页

硬连线

  • 优点：速度快，适合简单/规整的指令系统
  • 缺点：多输入/多输出巨大逻辑网络，复杂 ISA 难实现/维护，灵活性差

微程序的基本思想

  • 仿照程序设计的方法，编制每个指令对应的微程序
  • 微程序 = 若干微指令组成的序列
  • 微指令 = 若干微命令（控制信号取值）的组合
  • 所有微程序放在 控存 (Control Storage, CS)（一个 ROM）里
  • 执行指令时：取出对应微程序的微指令，译码产生微命令 = 控制信号

微程序优点

规整、可维护、灵活，但速度慢（多一层 ROM 查表）。

微程序控制器的基本结构

PDF 第 92 页

方框逐个认

  • μPC：微程序计数器，指向 CS 中要执行的下条微指令
  • CS (控制存储器)：ROM，存所有微程序
  • μIR：微指令寄存器，存"正在执行的微指令"
  • 译码器：把 μIR 译成控制信号（微命令）
  • 起始和转移地址发生器：根据 IR（机器指令）和条件码决定下条微指令的 μPC

工作循环

  1. 取出一条机器指令到 IR
  2. 起始地址发生器算出该指令的微程序首址 → μPC
  3. 每个时钟：CS[μPC] → μIR → 译码 → 控制信号 → 数据通路执行
  4. 顺序时 μPC+1；遇分支微指令时由转移地址发生器决定
固件 (Firmware) 一词的由来：最初指固化在 ROM 里的微程序——"用软件实现的硬件部件"。现在通俗理解为"ROM 中固化的软件"。

第一个问题：微指令格式（水平 vs 垂直）

PDF 第 94 页

水平型微指令

  • 思想：相容微命令尽量多地塞进一条微指令
  • 优点：微程序短，并行性高，速度快
  • 缺点：微指令长，编码空间利用低，编制困难

垂直型微指令

  • 思想：一条微指令只控制 1~2 个微命令（类似机器指令）
  • 优点：微指令短，编码紧凑，编制容易
  • 缺点：微程序长、无并行、慢
口诀：水平型面向内部控制逻辑（贴近硬件），垂直型面向算法描述（贴近程序员思维）。
现代 CPU 微码用的是水平型偏多——硬件资源足够，速度优先。

处理器中的异常处理机制

检测到异常时的三步基本动作

  1. 关中断（"中断/异常允许"状态位清 0）：防止新异常破坏现场
  2. 保护断点和程序状态：
    • PC 存入栈或专门的"断点寄存器"（RISC-V 用 mepc/sepc，MIPS 用 EPC，x86 压栈）

带异常处理的数据通路设计

在原多周期 CPU 上加什么

  • EPC 寄存器（32 位）：存放断点 PC。注意——故障类异常断点是正在执行的指令地址（需 PC-4，因为 PC 已被取指 +4），自陷/中断类断点是下条指令地址（直接送 PC）。
  • Cause 寄存器（32 位）：记录异常原因。假设两种：非法指令 (Cause=1)，溢出 (Cause=2)。
  • EPCWr, CauseWr：写使能信号
  • CauseMUX：选 1 还是 2 写入 Cause
  • PCMUX：在原有 PC 输入端再加一个 MUX，让"异常处理程序入口地址"（如 0x10000）也能写入 PC

这些新部件的加入不改变原数据通路骨架，只是多了几个寄存器和 MUX。

第五讲：流水线数据通路和控制。从直觉到设计到性能。

背景：吞吐率 vs 延迟（流水线的"卖点"）

补充背景 · 进入 p107 之前

两个相关但不同的概念

  • 延迟 (Latency)：单一任务从开始到结束的时间
  • 吞吐率 (Throughput)：单位时间能完成多少任务

流水线的关键洞察

流水线不能缩短单个任务的延迟，但能大幅提高吞吐率——因为多个任务可以"重叠"进行，每个阶段被不同任务复用。

类比：洗衣店

洗一批衣服 = 洗 30min + 烘 40min + 叠 20min = 90 min（不可压缩）。
但如果有 4 批衣服：

  • 串行：4 × 90 = 360 min
  • 流水：30 + 4×40 + 20 = 210 min（提速接近 3 倍）

第一批仍然要等 90 min，但后续每 40 min 就完成一批。延迟没变，吞吐率提高。

CPU 流水线的对应：一条指令仍要 5 个时钟才能"出口"，但每个时钟都有一条新指令进入流水线，CPI 接近 1。流水线的"40 min"对应"最长一段的延迟"——所以各阶段越均衡，效率越高。

单周期模型 vs 流水模型 性能比较

每个阶段所花时间

  • 取指：2 ns
  • 寄存器读：1 ns
  • ALU 操作：2 ns
  • 存储器读：2 ns
  • 寄存器写：1 ns

Load 总时间 = 2+1+2+2+1 = 8 ns（最长指令）。

单周期模型

  • 时钟周期 = 8 ns（被 lw 拖死）
  • N 条指令时间 = 8N ns

流水线模型

  • 时钟周期 = 最长阶段 = 2 ns
  • 每条指令延迟 = 2 ns × 5 = 10 ns（比单周期还长 2 ns！）
  • 但 N 条指令总时间 = (4 + N) × 2 ns（前 4 个时钟流水线填充，之后每拍一条指令完成）
  • N 很大时，比串行提速约 4 倍。若各阶段都恰好 2 ns，则满 5 倍。
反直觉关键：流水线单条指令延迟更长（要等满 5 拍 × 2ns），但吞吐率高得多——平均每拍出一条指令，CPI 接近 1。

流水线指令集的设计

PDF 第 114 页

什么样的 ISA 适合流水线？

  1. 指令长度一致：简化取指和译码。RV32I 都 32 位 → PC+4 万事大吉；x86 长度 1~17 字节 → 取指部件复杂到爆炸。
  2. 格式少，源寄存器位置固定：在未知指令类型时就能开始读寄存器。RV32I 的 rs1/rs2 位置在 R/I/S/B 格式里都固定，可以提前并行读。
  3. 只有 load/store 可访存：算术指令和访存分阶段——保持流水线规整。x86 的 ALU 指令可以直接操作内存，导致单条指令要算地址+访存+运算，无法塞进一个 EX 段。
  4. 内存"对齐"存放：访存少跨界，简化 Mem 段。
RISC-V 设计哲学的回报全部在这里兑现：简单、规整、一致 → 流水线友好 → 高性能。这是 RISC 比 CISC 在 90 年代崛起的核心原因。

• 每周期一条指令进入流水线
• 每周期一条指令完成

含 R-type 和 Load 的混合流水线 · 暴露结构冒险

PDF 第 117 页 · 第一个问题

问题

R-type 4 阶段，Load 5 阶段。把它们交错放：

第 7 周期出现：Load 在 Wr（第 5 阶段），后面紧跟的 R-type 在 Wr（第 4 阶段）—— 两条指令同时想写寄存器。同一个写口被两条指令同时使用 = 资源冲突。

这就是结构冒险 (Structural Hazard)

同一功能部件同一时刻被多条指令占用。本来流水线的前提就是"每个阶段被不同指令同时占用"，但每个阶段对应各自独立的资源。一旦不同阶段的指令想抢同一物理资源，就崩了。

流水线设计的两条铁律

① 每个功能部件每条指令只能用 1 次（写口不能用两次）
② 每个功能部件必须在相同的阶段被使用（写口总在第 5 阶段被使用）。

怎么实现规则 ②？让 R-type 的写也推迟到第 5 阶段——下页方案。

R-type 的 Wr 操作延后一个周期

PDF 第 118 页 · 经典解法

方法

在 R-type 的 Exec 后插入一个空的 Mem 段 (NOP)，让 Wr 推到第 5 段。所有指令现在都是 5 段（IF/ID/EX/M/WB），Wr 永远在第 5 段发生——规则 ② 满足。

代价

R-type 多了一个空操作周期，单条指令延迟从 4 变 5。但吞吐率不变（仍然每拍出一条），实际开销很小。

设计取舍：用"小幅增加单条指令延迟"换"流水线规整、易于实现"。这是流水线设计中常见的权衡——把稀有的复杂性硬塞进规则化框架，让公共情况快速。

  3. Exec：算地址 (rs1 + SEXT(imm12))
  4. Mem：写数据存储器
  5. Wr：空（NOP）— Store 不写寄存器，加这一段只是为对齐

Store 也跑满 5 阶段，第 5 阶段啥都不干。流水线规整 > 节省 1 个 NOP 周期。

Beq 的四个阶段

PDF 第 120 页

beq 真正需要的阶段

  1. Ifetch：取指、PC+4
  2. Reg/Dec：读 rs1, rs2, 译码
  3. Exec：① ALU 算 rs1-rs2（比较）② Adder 算转移地址
  4. Mem：如果 Zero=1，把目标地址写到 PC
  5. Wr：空 NOP

分支决定在 Mem 段才完成——这是控制冒险的根源（p125 详讲）。比 beq 后面紧跟的指令此时已经被取出来 2-3 个了！

流水线中的"竞争"问题

PDF 第 121 页

多周期中怎么解决 Addr/WrEn 竞争

多周期中：第 N 周期末让 Addr 有效，第 N+1 周期让 WrEn 有效 — 保证 Addr 先到。

流水线为啥不能这么做

因为每个周期都必须能写（Register / Memory），不能像多周期那样把 WrEn 推迟一拍——会卡住整条流水线。

解决方案：读写口物理分离（寄存器组 2 读 1 写口可独立工作）；指令存储器和数据存储器分开（哈佛架构）。结构冒险通过"加资源"消除。

  • 指令执行可以像洗衣服一样流水化，吞吐率提 N 倍，单条延迟不变
  • 流水段数 = 最复杂指令的步骤数；每段宽度 = 最复杂阶段时间

流水线三种冒险（Hazard）总览

PDF 第 124 页 · 最重要的分类

Hazard 的定义

流水线遇到无法正确执行后续指令，或执行了不该执行的指令的情况。

三种冒险

名称现象典型对策
结构冒险
(hardware resource conflicts)		同一部件被多条指令同时使用① 每部件每条指令只能用 1 次
② 只能在特定阶段使用
③ 设置多个独立部件（IM/DM 分开）
数据冒险
(data dependencies)		后面指令用到前面指令的结果，但前面还没写完① 转发 Forwarding/Bypassing
② Load-use 必须阻塞 1 拍
③ 编译器优化指令顺序
控制冒险
(branch hazards)		分支/异常改变 PC，但后续指令已被取入流水线① 静态/动态分支预测
② 延迟分支（编译器调度）
这三种是流水线所有问题的总和。剩下整个第六讲就是讲怎么对付它们。

背景：转发 / 阻塞 / 预测 三个核心机制

补充背景 · 进入 p125 之前

转发 (Forwarding / Bypassing)

前一条指令的结果还没写回寄存器，但已经在 EX 段算完了——可以从 EX 段输出端直接接线到后一条指令的 EX 输入端。不等写回也能用。

关键：转发是用"硬件加一根线"换"避免气泡"。对 ALU 结果型的依赖完全有效。

阻塞 (Stall) — 又叫"气泡 Bubble"

实在没法避免冲突时，让后面指令暂停一拍（IF/ID 段冻结，EX 段进 NOP）。流水线里出现一个"空泡"。CPI 上升。

典型场景：Load-use 冒险。lw 在 Mem 段才能拿到数据，下条指令立即要用——转发都来不及，必须 stall 1 拍。

预测 (Prediction)

分支指令在 Mem 段才知道走哪，但流水线已经取了 2-3 条后续指令——提前猜，猜对则继续，猜错就把流水线里的错指令清空（flush）。

  • 静态预测：总是预测"不跳"（针对前向跳转），准确率 65-85%
  • 动态预测：根据历史调整，准确率可达 90%+

三机制一图记

[ 转发 ] ← 修补"数据还没到位"的依赖
[ 阻塞 ] ← 实在没法时的兜底
[ 预测 ] ← 分支不可知时的赌博

Control Hazard 的解决方法

PDF 第 125 页

四种解决方法

  1. 硬件阻塞：在分支后插 3 个 NOP（硬件把后续指令清 0）。简单但效率低。
  2. 软件插 NOP：编译器在分支指令后塞 3 个 NOP。同样低效。
  3. 分支预测：
    • 静态：总是预测 not taken。+启发式（循环底部跳总成立）。准确率 65-85%
    • 动态：根据历史调整。准确率 90%+
  4. 延迟分支 Delayed Branch：把与分支无关的前置指令挪到分支指令后面执行。编译器在编译期完成调度，分支后的"延迟槽"被有用工作填满。
关键点：流水线控制必须确保被错误预测的指令的执行结果不能生效，并且能从正确的分支地址处重启流水线。这就是 "flush" 的硬件机制。

异常/中断也是一种控制冒险（突然改变 PC），处理逻辑类似。

三种处理器实现方式的综合性能比较

PDF 第 126 页 · 大数字例

条件假设

指令混合 (假设)：Load 25%, Store 10%, R-Type/ALU 52%, Branch 11%, Jump 2%。每个功能部件最长 200 ps，单周期 lw 总 600 ps。

三种方式的 N 条指令总时间

单周期

时钟周期 = 600 ps（被 lw 决定），N 条指令 = 600N ps

多周期

时钟周期 = 200 ps（最长功能部件），CPI 计算：

CPI = 5×0.25 + 4×0.10 + 4×0.52 + 3×0.11 + 3×0.02 = 4.12

N 条指令 = 4.12 × 200 × N = 824N ps

流水线

时钟周期 = 200 ps。CPI 计算（要考虑冒险造成的额外周期）：

  • Load：load-use 依赖时多 1 拍，否则 1 拍 → 平均 1.5 拍
  • Store/ALU：1 拍
  • Branch：预测对 1 拍，错则 2 拍 → 平均 1.25 拍
  • Jump：2 拍（要等译码段完才能算目标地址）
CPI = 1.5×0.25 + 1×0.10 + 1×0.52 + 1.25×0.11 + 2×0.02 = 1.17

N 条指令 = 1.17 × 200 × N = 234N ps

三者比较

总时间相对
单周期600N慢 1×
多周期824N更慢 (!)
流水线234N快约 2.5×
反直觉：多周期反而比单周期慢！因为这里指令混合中长指令比例不够高，多周期的 CPI 涨幅吃掉了时钟缩短的好处。这说明多周期的优势取决于工作负载。

流水线无悬念赢——同时享受到"短时钟周期" + "接近 1 的 CPI"。

三种实现的视觉对比

  • 单周期：每个时钟（很长）做完一条指令；Store 比 Load 短但要等同一个时钟 → 浪费
  • 多周期：时钟短了 5 倍，但每条指令仍按"段数 × 时钟"顺序执行，没有重叠

本章 5~7 讲总结 3：流水线的局限 + 高级技术

PDF 第 128 页

流水线的局限

  • 并非每条指令都有相同段数
  • 并非每段都一样长（必须对齐到最长）
  • 流水线深度越大，寄存器额外开销比例越大
  • 遇到资源冲突、数据相关、控制相关时发生冒险

更高级的流水线技术

  • 超流水线 (Super-Pipelined)：流水段更多、更细（如 Intel Pentium 4 高达 20+ 段），主频更高
  • 多发射流水线 (Multi-Issue)：同时发射多条指令进同一流水线
    • 静态多发射 (VLIW)：编译器把多条无依赖的指令打包发射
    • 动态多发射 (超标量 Superscalar)：硬件动态分析依赖、调度发射
现代高性能 CPU（Intel/AMD/Apple M 系列）都是超标量 + 乱序执行 + 深流水线的混合体。本章只讲了最简单的 5 段静态流水线，是入门基础。

本章 5~7 讲总结 4：三种冒险对策汇总

PDF 第 129 页 · 必背

结构冒险（资源冲突）

  • 规定每部件每条指令只能用 1 次
  • 规定每部件只能在特定阶段用
  • 指令存储器 IM 和数据存储器 DM 分开（哈佛架构）

数据冒险（数据相关）

"后面指令的源操作数 = 前面指令的目的"

  • 软件阻塞：编译器在数据相关指令前插 nop
  • 硬件阻塞 (stall)：硬件在特定流水段插"气泡"
  • 转发 Forwarding/Bypassing：把前面执行结果直接传到后面指令的相应流水段（最常用！）
  • 对 Load-use：阻塞+转发组合
数据冒险按数据来源分两种：
① 前面指令结果在 EX 末就能拿到（如 add）→ 完全可用转发解决
② 前面指令结果在 Mem 末才能拿到（Load）→ 至少 stall 1 拍，再转发

本章 5~7 讲总结 5：控制冒险细化

PDF 第 130 页

控制冒险的定义

返回指令、分支指令等可能改变顺序增量的 PC 值。由于获取转移目标地址要时间，目标地址产生前已有错误指令进入流水线 → 控制冒险。

四种对策汇总

  • 软件阻塞：编译器在控制相关指令后插 nop
  • 硬件阻塞：硬件插气泡停顿若干拍，直到 PC 确定
  • 分支预测：
    • 静态：总猜 not taken
    • 动态：根据历史，可达 90%+ 准确
  • 延迟分支：编译器把前面的无关指令挪到分支指令后执行（延迟槽）

异常/中断是特殊的控制冒险，流水线中实现具有挑战性（要在正确的指令边界精确触发）。

组成指令功能的四种基本操作 + RTL

PDF 第 131 页 · 全章收束

无论 ISA 多复杂，指令的基本操作只有 4 种

  1. 读取某主存单元到寄存器（取指、取数）
  2. 把寄存器存到主存（存结果）
  3. 寄存器之间或寄存器到 ALU 传送数据（取数、存结果）
  4. 算术或逻辑运算（PC+1、算地址、运算）

RTL（寄存器传送语言）回顾

  • R[r]：寄存器 r 的内容
  • M[addr]：主存单元 addr 的内容
  • PC：直接表示其内容；M[PC]：PC 所指内存单元内容
  • SEXT[imm] / ZEXT[imm]：符号/零扩展
  • ←：传送，源在右，目的在左

例

R[x8] ← M[R[x9]+SEXT(imm12)] = 把"x9 的内容 + 立即数（符号扩展）"作为内存地址，取出内容送到寄存器 x8。这正是 lw x8, x9, imm12。
