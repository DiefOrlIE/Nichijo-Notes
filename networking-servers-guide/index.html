计算机网络与服务器
从零到搭起你的网站和代理

这份教程用两个真实项目当主线——搭一个动态个人网站、搭一套自用的个人代理——把计算机网络和服务器的核心知识从最底层讲清楚。看得懂、用得上、将来能往深里挖。

读者 几乎零网络基础 风格 原理 + 实操,由浅入深 主线 动态网站 · 个人代理 建议 边读边动手

PART 0怎么用这份教程

你现在大概是这样:会用浏览器,知道"上网"这回事,但 IP、DNS、端口、服务器这些词,听过、说不清。没关系——这份教程就是为这个起点写的。每个概念都会先用生活里的类比建立直觉,再补上技术细节,最后落到"在这两个项目里它具体是什么"。

一条核心主线

整份教程其实在回答一个问题:"你电脑上的一段数据,怎么准确地送到地球另一端的另一台电脑,再原样拿回来?" 网站是"你去访问别人的服务器",代理是"你让自己的服务器替你去访问别人"。两件事用的是同一套底层知识,学一遍,两个项目都通了。

三种读法,按你的状态选

  • 系统学(推荐):从第 1 部分顺着读到第 6 部分。前 3 部分是地基(网络原理 + 服务器基础),后面两个项目会反复用到。
  • 带着目标学:急着搭网站就先读 1、2、3,再跳到第 4 部分;急着搭代理就读 1、3,再跳到第 5 部分。遇到不懂的名词回前面查。
  • 当手册查:左侧目录随时跳转。附录有术语表、命令速查和排错思路。

PART 1先建立心智模型:网络到底是怎么回事

在背任何术语之前,先把整幅图装进脑子。这一部分讲完,你再看任何网络知识都有地方"挂"。核心就一句话:网络是一套让两台机器可靠交换数据的规则。下面把这套规则拆开看。

1.1 输入网址按下回车,那一瞬间发生了什么

这是理解网络最好的入口。假设你在浏览器输入 www.example.com 然后回车,看似一瞬间页面就出来了,背后其实跑完了一条很长的流水线:

  1. 查地址(DNS):www.example.com 是给人记的名字,机器之间只认数字地址(IP)。浏览器先问"DNS 服务器":这个名字对应哪个 IP?
  2. 接通线路(TCP):拿到 IP 后,你的电脑和服务器先"握手"建立一条可靠连接,确认双方都在线、能收发。
  3. 上锁(TLS):如果是 https,双方再协商一把只有彼此知道的密钥,之后所有内容都加密,中间人偷看到的是乱码。
  4. 提要求(HTTP):连接建好后,浏览器发一句结构化的话:"GET 首页"。
  5. 给东西(响应):服务器回一坨 HTML 文本。
  6. 画出来(渲染):浏览器把 HTML/CSS/JS 解析成你看到的页面。
类比:寄一封挂号信

DNS 是"查收件人地址";TCP 握手是"先打电话确认对方在家、能签收";TLS 是"用只有你俩懂的暗语写信";HTTP 请求是"信里写的内容";响应是"对方回的信"。整套网络协议,本质就是一份大家都遵守的"邮政规则"。

1.2 为什么要"分层":网络的分层思想

上面每一步用的是不同的"协议"(协议 = 双方约定好的规则)。它们被组织成分层结构:每一层只干自己的事,把上一层的数据包好交给下一层,像套娃。这样做的好处是各层可以独立替换——换 WiFi 还是网线(底层变了),你的浏览器(高层)完全不用改。

术语:OSI 七层 vs TCP/IP 五层

课本常讲 OSI 七层模型(物理、数据链路、网络、传输、会话、表示、应用),那是教学用的理论参考。实际互联网跑的是 TCP/IP 模型,通常按上图划成 4 或 5 层。入门阶段记住应用层 / 传输层 / 网络层 / 链路层这四层和它们各自管什么,就足够用很久了。

1.3 IP 地址:网络世界的门牌号

互联网上每一台要被找到的机器,都得有一个IP 地址。它就是门牌号。目前两种格式并存:

  • IPv4:形如 93.184.216.34,四段 0–255 的数字。总共约 43 亿个,早就不够分了。
  • IPv6:形如 2606:2800:220:1:248:1893:25c8:1946,数量大到几乎用不完,正在逐步普及。

公网 IP 与私网 IP

这是初学者最容易混的一点。你家里每台设备的 IP(比如手机显示 192.168.x.x)是私网 IP,只在你家这张局域网里有效,出了家门没人认。真正在互联网上代表"你家"的,是路由器从运营商那拿到的一个公网 IP。

类比:公司总机 + 分机

公网 IP 像公司对外的总机号码(全世界唯一、能打进来);私网 IP 像公司内部的分机号(1001、1002,出了公司没用)。下一节的 NAT 就是那台"总机转接"的设备。

动手:看看自己的 IP

查私网 IP(本机在局域网里的地址):Windows 上 Win+R 输入 cmd,运行 ipconfig;找 IPv4 地址那行。

查公网 IP(你家对外的地址):浏览器访问 ip.sb 或搜索"我的 IP"。把两个数字对比一下,你就理解"里外两个地址"了。

1.4 端口:一台机器上的多个窗口

IP 把数据送到了正确的机器,但一台机器上同时跑着很多程序——浏览器、QQ、网游、网站服务……数据到底交给哪个程序?靠端口号(0–65535 的整数)区分。IP 定位机器,端口定位机器上的具体程序。

类比:大楼 + 房间号

IP 是大楼地址,端口是房间号。93.184.216.34:443 意思是"那栋楼的 443 房间"。一些端口是约定俗成的"固定铺面":

端口常见用途你会在哪用到
80HTTP(网页,明文)网站项目
443HTTPS(网页,加密)网站项目 + 代理伪装
22SSH(远程登录服务器)两个项目都要
53DNS(域名解析)底层自动用
3306 / 5432MySQL / PostgreSQL 数据库网站后端

记住 22 / 80 / 443 这三个就够开局了:登录服务器走 22,网站走 80/443。

1.5 DNS:把域名翻译成 IP 的电话簿

没人愿意背 93.184.216.34,大家记的是 example.com。DNS(域名系统)就是那本全球电话簿,负责把好记的域名翻译成机器要用的 IP。这个翻译过程叫域名解析。

解析是逐级问出来的,像查一本分层目录:

这对两个项目都很关键

网站项目:你买了域名后,要在域名商后台加一条 DNS 记录,把 你的域名 → 你服务器的 IP 绑起来,别人才能用域名访问你。

代理项目:高级伪装方案会用到域名 + DNS,让流量看起来像在访问一个普通网站。

最常用的两种记录:A 记录(域名 → IPv4)、CNAME(域名 → 另一个域名,做别名)。

1.6 TCP / UDP / 数据包:东西是怎么一段段传过去的

数据在网上不是一股脑流过去的,而是被切成一个个数据包(packet)分别发送,到对面再重新拼起来。负责"怎么切、怎么拼、丢了怎么办"的,是传输层的两个协议:

TCPUDP
核心特点可靠、有序、会重传快、简单、不保证送到
类比挂号信(要签收回执)明信片(扔进邮筒就不管了)
适合网页、文件、登录——错一个字节都不行直播、语音、游戏——快比全更重要
你的项目网站全程用 TCP新一代代理协议(如 Hysteria)用 UDP 提速

TCP 的"三次握手"

TCP 在传数据前,要先确认双方都准备好,这就是三次握手——你会反复在各种文档里见到这个词,它其实很简单:

1.7 路由与 NAT:包是怎么找到地球另一端的

你的数据包从家里出发,要经过运营商、骨干网、海底光缆、对方运营商……十几二十"跳"才能到目标服务器。每一跳上都有一台路由器,它干的事只有一件:看包上的目标 IP,查自己的"路由表",决定"下一站往哪扔"。无数路由器接力,包就一步步逼近目标。

动手:亲眼看包走过的路

命令行运行 tracert example.com(Windows)或 traceroute(Linux/Mac),会列出你的包经过的每一跳路由器。再用 ping example.com 看往返要多少毫秒(延迟)。这俩是日后排查"网络不通/很慢"的第一手工具。

NAT:让一个公网 IP 养活全家设备

前面说你家所有设备共用一个公网 IP,实现这点的就是路由器上的 NAT(网络地址转换)。你发出去的包,源地址会被路由器从私网 IP 改写成公网 IP;回来的包再改回来,准确送到当初那台设备。

NAT 解释了一个关键现象

因为有 NAT,外部默认无法主动连到你家的某台设备(它没有独立公网地址,藏在路由器后面)。这就是为什么你的网站和代理都得放在一台有公网 IP 的服务器(VPS)上——它能被全世界直接访问到,而你家的电脑通常不能。这条直接引出第 3 部分。

第 1 部分小结(务必记住这几句)
  • IP 定位机器,端口 定位机器上的程序,DNS 把域名翻译成 IP。
  • 数据被切成包,TCP 保证可靠送达,路由器接力转发。
  • 网络分层,各层套娃式打包;你写网站只碰最上面的应用层,但出问题要会往下看。
  • 你家设备藏在 NAT 后面,所以需要一台有公网 IP 的服务器来对外提供服务。

PART 2Web 的语言:HTTP 与 HTTPS

第 1 部分搞定了"数据怎么送到对面"。这一部分讲"送的内容长什么样"。浏览器和服务器之间说的话遵循 HTTP 协议——这是你做网站时天天打交道的东西,做代理时也要靠它来伪装。

2.1 请求与响应:HTTP 的全部骨架

HTTP 是个简单的一问一答模型:客户端(浏览器)发一个请求,服务器回一个响应,然后这次对话就结束了。一个请求长这样:

# 请求:浏览器发给服务器的原始文本
GET /about.html HTTP/1.1          # 方法 + 路径 + 协议版本
Host: www.example.com             # 要访问哪个域名
User-Agent: Mozilla/5.0 ...       # 我是什么浏览器
Accept: text/html                 # 我想要什么格式
                                  # ← 这里有个空行,下面是请求体(GET 通常没有)

服务器回的响应,结构对称:

# 响应:服务器回给浏览器
HTTP/1.1 200 OK                   # 状态码:成功
Content-Type: text/html           # 我给你的是 HTML
Content-Length: 1270              # 内容多长
                                  # ← 空行
<!DOCTYPE html><html>...</html>    # 响应体:真正的网页内容

请求方法:你想干什么

方法含义例子
GET获取数据(只读)打开一个网页、查列表
POST提交新数据注册、发评论、上传
PUT / PATCH修改已有数据编辑个人资料
DELETE删除数据删一条评论

你的动态网站后端,本质就是给这几种方法各写一段处理逻辑。第 4.3 节会专门讲。

状态码:服务器的回复是好是坏

范围含义常见
2xx成功200 OK、201 Created
3xx重定向(去别处)301 永久跳转、302 临时
4xx你(客户端)错了404 找不到、401 没登录、403 没权限
5xx服务器自己崩了500 内部错误、502 网关错误
这些数字你会天天见

部署网站时,502 Bad Gateway 通常意味着 Nginx 转发的后端程序没起来;404 是路径写错或文件没传上去;500 是你后端代码报错了。看到几百几,先判断是自己错(4xx)还是服务器错(5xx),排错方向立刻清晰。

请求头 / 响应头:附带的元信息

上面那些 Host:、Content-Type: 叫头(header),是关于这次请求/响应的"说明标签"。它们控制缓存、身份、内容类型、跨域等等。现在不用背,知道"头是一堆 key: value 的元信息"即可。

2.2 一个关键难题:HTTP 是"无状态"的

HTTP 有个反直觉的设定:每次请求都是独立的,服务器默认不记得你上一次是谁。这叫无状态。可是网站明明能"保持登录",怎么做到的?靠三种机制给无状态的 HTTP "外挂记忆":

机制怎么工作类比
Cookie服务器发一小段数据让浏览器存着,之后每次请求自动带上游乐园门口盖的手印章
Session服务器端存你的登录状态,Cookie 里只放一个 session ID 当钥匙寄存柜:你拿号牌,东西在柜子里
Token(如 JWT)登录后发你一张带签名的"令牌",你每次请求放在头里;服务器验签不用查库演唱会的电子票,扫码即验
做网站时你要选一种

"评论 / 登录"功能的核心就是这个。传统服务端渲染网站常用 Session + Cookie;前后端分离 / 手机 App 常用 Token(JWT)。第 4 部分会落地。现在记住:登录态 = 服务器靠每次请求带的那点凭证认出你。

2.3 HTTPS 与 TLS:为什么要加密,怎么加密

普通 HTTP 是明文传输——你的密码在路上,任何一跳路由器、同一个 WiFi 下的人都可能看到。HTTPS = HTTP + TLS 加密,给整条通道上锁。今天所有正经网站都必须上 HTTPS(浏览器会把纯 HTTP 站标成"不安全")。

TLS 要同时解决两个问题:

  1. 加密:内容变乱码,中间人看不懂。
  2. 身份验证:确认对面真是 example.com,而不是冒充的钓鱼站——靠证书。
术语:证书、CA、Let's Encrypt

证书是一份"由权威机构盖章、证明某域名归某服务器所有"的文件。盖章的权威机构叫 CA(证书颁发机构)。浏览器内置了一批信任的 CA,所以能验证证书真假。
Let's Encrypt 是一个免费、自动化的 CA,你的网站项目就用它来一键签发 HTTPS 证书(见 4.7 节)。代理项目的高级伪装也会用到证书。

类比:公钥 / 私钥这把"特殊的锁"

TLS 用到非对称加密:一对钥匙,公钥负责"锁"、私钥负责"开",公钥可以随便公开。任何人都能用你公开的公钥给你加密,但只有握着私钥的你能解开。证书里装的就是服务器的公钥。这套思想后面 SSH 登录(3.3)、代理加密(5.4)还会反复出现,先有个印象。

2.4 动手:用浏览器开发者工具看真实的网络通信

讲再多不如亲眼看一次。打开任意网站,按 F12 打开开发者工具,切到 Network(网络) 面板,刷新页面:

照着看一遍
  1. 左边列出了这个页面发起的所有请求(HTML、CSS、图片、接口……一个页面往往有几十个)。
  2. 点任意一条,右边能看到:请求的 URL、方法、状态码;Headers(请求头/响应头,找找 Cookie 和 Content-Type);Response(服务器返回的原始内容)。
  3. 切到 Console(控制台) 能跑 JavaScript;切到 Application 能看到本站存的 Cookie。

这一步把前两部分的抽象概念全部"实体化"了:你刚学的请求/响应/状态码/头/Cookie,现在都摆在眼前。这是你以后调试网站最常用的工具,务必玩熟。

第 2 部分小结
  • HTTP 是一问一答:请求(方法+路径+头+体)→ 响应(状态码+头+体)。
  • 状态码分四档:2 成功 / 3 跳转 / 4 你错 / 5 服务器错。
  • HTTP 无状态,靠 Cookie / Session / Token 维持登录。
  • HTTPS = HTTP + TLS:先用证书验身份,再协商密钥加密。免费证书用 Let's Encrypt。

PART 3服务器与 Linux 基础

两个项目都要落到一台有公网 IP、24 小时开机的服务器上。这一部分讲清楚:服务器是什么、为什么是 Linux、怎么远程登录它、怎么在上面敲命令干活。这是从"在自己电脑上写代码"跨到"让全世界能访问"的关键一跳。

3.1 服务器、VPS、云,到底是什么

服务器(server)不是一种特殊的机器,而是一个角色:任何一台持续运行、等着别人来连接并提供服务的电脑,都是服务器。你手头这台电脑也能当服务器,但它有两个硬伤:不是 24 小时开机,而且(回忆 1.7 节)藏在 NAT 后面,外面连不进来。

所以现实做法是租一台云服务器。最常见的形态是 VPS(虚拟专用服务器):云厂商把一台物理大服务器切成很多份,每份给你一个独立的操作系统、固定公网 IP 和 root 权限,你像用一台真机一样用它。

项目选什么样的 VPS大致价位
动态网站国内或就近地区;若面向国内访客且用自有域名,需备案¥20–60/月起
个人代理海外 VPS(如日本、新加坡、美西),线路质量优先¥30–60/月
术语:常见云厂商

国内:阿里云、腾讯云、华为云。海外:Vultr、DigitalOcean、Linode(Akamai)、AWS Lightsail、Hetzner、搬瓦工(BandwagonHost)等。新手买 VPS 看四个指标:地区/线路、CPU与内存、流量额度、价格。代理用途尤其看"流量"和"线路"。

关于备案(国内网站绕不开)

在中国大陆的服务器上、用域名对外提供网站服务,法规要求先做 ICP 备案(通过云厂商提交,实名、需时约 1–3 周)。想跳过备案练手,可以先用海外服务器 + 域名直连,或用 GitHub Pages / Vercel 这类托管平台起步。这点务必先了解清楚,免得买完才发现卡住。

3.2 Linux 入门:为什么服务器几乎都跑 Linux

绝大多数服务器跑的是 Linux(一种免费、开源、稳定、省资源的操作系统),最常见的发行版是 Ubuntu 和 Debian。它没有图形界面,全靠命令行操作——一开始会不习惯,但命令行恰恰是服务器运维的"母语",学会受益十年。

三个必须建立的概念

  • 一切皆文件:配置、设备、甚至运行中的进程,在 Linux 里大多表现为文件路径。改配置 = 编辑某个文件。
  • 目录树:从根 / 开始的一棵树。关键几个:/etc(配置)、/var/log(日志)、/home(用户文件)、/usr(程序)、/root(管理员家目录)。
  • 用户与权限:root 是超级管理员(权力无限,也最危险);普通用户权限受限。临时借用管理员权限,在命令前加 sudo。
权限那一串 rwx 怎么读

ls -l 会看到 -rwxr-xr-- 这种。r=读、w=写、x=执行,三组分别对应文件主人 / 同组 / 其他人。chmod 改权限、chown 改归属。配置代理和网站时常因为"权限不对"出问题,知道这个就能自查。

3.3 SSH:安全地远程登录服务器

VPS 在云端,你怎么操作它?靠 SSH——一个加密的远程登录协议(走 22 端口)。在你自己电脑的终端里敲一行,就进了服务器的命令行:

# 用密码登录(root 用户 + 服务器公网 IP)
ssh root@123.45.67.89

密码登录能用,但不安全也不方便(会被暴力破解、每次要输密码)。正确做法是用 SSH 密钥对登录——回忆 2.3 节的公钥/私钥思想,这里正好用上:

# ① 在你自己电脑上生成一对密钥(一路回车即可)
ssh-keygen -t ed25519

# 生成两个文件:
#   ~/.ssh/id_ed25519       私钥 —— 死守在自己电脑,绝不外传
#   ~/.ssh/id_ed25519.pub   公钥 —— 可以公开,要放到服务器上

# ② 把公钥安装到服务器(之后免密登录)
ssh-copy-id root@123.45.67.89
# Windows 若没有该命令,可手动把 .pub 内容追加到
# 服务器的 ~/.ssh/authorized_keys 文件里
类比:私钥是钥匙,公钥是锁

你把"锁"(公钥)装到服务器门上,"钥匙"(私钥)只有你有。登录时服务器用锁出一道只有对应钥匙能解的题,你的电脑用私钥解开,证明"是你"。全程私钥不离开你的电脑,比密码安全得多。私钥泄露 = 别人能登你服务器,务必保管好。

3.5 进程与服务:让程序在后台一直跑(systemd)

你的网站后端、你的代理程序,都需要开机自启、崩了自动重启、关掉 SSH 也继续跑。现代 Linux 用 systemd 管这件事,被它管理的长期运行程序叫一个服务(service)。

# systemctl:管理服务的总开关(以 nginx 为例)
sudo systemctl start nginx      # 启动
sudo systemctl stop nginx       # 停止
sudo systemctl restart nginx    # 重启(改完配置常用)
sudo systemctl status nginx     # 看运行状态 + 最近日志(排错第一站)
sudo systemctl enable nginx     # 设为开机自启
为什么这个很重要

新手常犯的错:在 SSH 里直接 python app.py 把网站跑起来,一关终端网站就挂了。正确姿势是把程序注册成 systemd 服务(或用 Docker、或用 pm2 这类进程管理器),让它脱离你的终端独立运行。第 4、5 部分都会这么做。

3.6 防火墙:只开该开的门

服务器暴露在公网上,每天都有自动化扫描在试探各个端口。防火墙的原则很简单:默认全关,只放行你确实在用的端口。Ubuntu 上最简单的工具是 ufw:

# 放行必要端口,再开启防火墙(顺序别反!先放 22 再 enable)
sudo ufw allow 22        # SSH,不放行你会把自己锁在外面
sudo ufw allow 80        # HTTP
sudo ufw allow 443       # HTTPS
sudo ufw enable          # 启用
sudo ufw status          # 查看规则
血泪坑:别把自己关在门外

开防火墙前务必先放行 22 端口,否则 enable 之后你自己的 SSH 也会被挡,直接失联(只能去云厂商后台的网页控制台救场)。这是新手最经典的事故之一。云厂商后台通常还有一层"安全组",逻辑相同,两边都要放行。

第 3 部分小结
  • 服务器是个角色;现实里租一台 VPS(有公网 IP、24h 开机、root 权限)。
  • 服务器跑 Linux,靠命令行操作;sudo 借管理员权限,权限模型是 rwx。
  • 用 SSH 密钥登录(私钥死守,公钥放服务器)。
  • 长期程序注册成 systemd 服务;防火墙默认全关、只放 22/80/443。

4.3 后端与 API:网站的大脑

后端是跑在服务器上的程序,负责:接收前端请求 → 校验身份 → 读写数据库 → 返回结果。前端和后端之间约定好的那套接口,叫 API。最主流的风格是 REST:用 HTTP 方法 + 路径表达"对什么资源做什么操作"。

前端想做的事对应 API(方法 + 路径)
看所有评论GET /api/comments
发一条评论POST /api/comments
删除第 5 条评论DELETE /api/comments/5
登录POST /api/login

用 Python 的 Flask 写,后端核心就这么几行(你是 AI 专业,Python 正好顺手):

# app.py — 一个最小后端
from flask import Flask, request, jsonify
app = Flask(__name__)

comments = []   # 先用内存列表,下一节换成数据库

# 前端 GET 过来:把所有评论给它
@app.route("/api/comments", methods=["GET"])
def list_comments():
    return jsonify(comments)

# 前端 POST 过来:新增一条评论
@app.route("/api/comments", methods=["POST"])
def add_comment():
    data = request.get_json()
    comments.append({"text": data["text"]})
    return jsonify({"ok": True}), 201   # 201 = 创建成功

if __name__ == "__main__":
    app.run(port=8000)

本机跑起来后,浏览器访问 http://127.0.0.1:8000/api/comments 就能看到后端的返回。前端那段 fetch 一调,前后端就接通了。

术语:后端技术选型(任选一个深入)

Python:Flask(极简,适合入门)、FastAPI(现代、自带接口文档、性能好)、Django(大而全)。Node.js:Express、Nest。Go:性能强,部署简单。
建议你从 Flask 或 FastAPI 起步——语言你熟,生态成熟,资料多。

4.4 数据库:让数据长期、可靠地存下来

上面把评论存在内存列表里,程序一重启就全没了。真正的数据要进数据库。两大流派:

关系型(SQL)非关系型(NoSQL)
代表PostgreSQL、MySQL、SQLiteMongoDB、Redis
结构像 Excel 表格,行列严格像 JSON,灵活
适合有明确关系的数据(用户、评论、订单)缓存、日志、结构多变的数据
你的网站首选 SQL,先用 SQLite 练手,上线换 PostgreSQL以后做缓存再说

用 SQL 描述"评论表",再增/查:

-- 建一张评论表(每条评论一行)
CREATE TABLE comments (
    id        SERIAL PRIMARY KEY,          -- 自增主键,每行唯一编号
    user_id   INTEGER,                     -- 谁发的(关联用户表)
    text      TEXT NOT NULL,               -- 评论内容
    created_at TIMESTAMP DEFAULT now()      -- 发表时间
);

INSERT INTO comments (user_id, text) VALUES (1, '第一条评论');  -- 增
SELECT * FROM comments ORDER BY created_at DESC;          -- 查
安全红线:SQL 注入

千万别把用户输入直接拼进 SQL 字符串(比如 "... text='" + 用户输入 + "'")。攻击者能输入特制内容篡改你的查询,这叫 SQL 注入。正确做法是用参数化查询(把值作为参数传入,由数据库驱动转义)或用 ORM。这是后端第一条要刻进肌肉记忆的安全准则。

术语:ORM

每次手写 SQL 很繁琐,ORM(对象关系映射,如 Python 的 SQLAlchemy)让你用写代码对象的方式操作数据库,它在背后生成安全的 SQL。入门可以先手写几条 SQL 理解原理,再上 ORM 提效。

4.6 部署上线:把网站搬到 VPS 上跑起来

这一步把"本机能跑"变成"全世界能访问"。完整链路:代码上传 → 装运行环境 → 用生产级方式启动后端 → Nginx 反向代理 → systemd 守护。

# 在 VPS 上(已用 SSH 登录)

# ① 装 Python 环境与依赖(以 Flask 为例)
sudo apt update && sudo apt install -y python3-pip python3-venv nginx
# 把代码传上来(git clone 你的仓库,或用 scp 上传)
cd /var/www/mysite
python3 -m venv venv && source venv/bin/activate
pip install flask gunicorn

# ② 用 gunicorn 以生产方式启动后端(比 app.run 更稳更快)
#    开 4 个工作进程,只监听本机 8000 端口(对外由 Nginx 转发)
gunicorn -w 4 -b 127.0.0.1:8000 app:app

接着配 Nginx 反向代理,把外面 80 端口的请求转发给本机 8000 的后端。编辑 /etc/nginx/sites-available/mysite:

server {
    listen 80;
    server_name hongzunyu.com www.hongzunyu.com;

    location / {
        proxy_pass http://127.0.0.1:8000;     # 转发给后端
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
# 启用配置并重载 Nginx
sudo ln -s /etc/nginx/sites-available/mysite /etc/nginx/sites-enabled/
sudo nginx -t              # 先测语法,没问题再重载
sudo systemctl reload nginx

最后把 gunicorn 注册成 systemd 服务(回忆 3.5),让它开机自启、崩溃重启。新建 /etc/systemd/system/mysite.service:

[Unit]
Description=My site backend
After=network.target

[Service]
WorkingDirectory=/var/www/mysite
ExecStart=/var/www/mysite/venv/bin/gunicorn -w 4 -b 127.0.0.1:8000 app:app
Restart=always

[Install]
WantedBy=multi-user.target
sudo systemctl daemon-reload
sudo systemctl enable --now mysite     # 启动并设为开机自启
sudo systemctl status mysite           # 确认在跑

到这一步,浏览器访问 http://hongzunyu.com 应该能看到你的网站了(还是 http,下一节加锁)。

4.7 HTTPS:给网站免费上锁(Let's Encrypt)

回忆 2.3 节,正经网站必须 HTTPS。用 certbot + Let's Encrypt 一条命令搞定,还能自动续期:

# 装 certbot 的 nginx 插件
sudo apt install -y certbot python3-certbot-nginx

# 自动签发证书 + 改好 Nginx 配置 + 设置 80 跳 443
sudo certbot --nginx -d hongzunyu.com -d www.hongzunyu.com

# 证书 90 天到期,certbot 会自动续期;可手动测试续期流程
sudo certbot renew --dry-run

跑完访问 https://hongzunyu.com,地址栏出现 🔒。别忘了在防火墙和云安全组放行 443。

4.8 安全加固:上线后立刻要做的几件事

服务器一旦上公网,几分钟内就会有自动化扫描来敲门。基础加固清单(投入产出比最高的几条):

做什么为什么
禁用 SSH 密码登录,只留密钥挡掉绝大多数暴力破解(改 /etc/ssh/sshd_config 里 PasswordAuthentication no)
不用 root 日常操作,建普通用户 + sudo降低误操作和被入侵后的破坏面
防火墙只开 22/80/443减少暴露面(回忆 3.6)
装 fail2ban自动封禁反复尝试登录失败的 IP
定期 apt upgrade + 关注安全更新补已知漏洞
后端:参数化查询、校验输入、密码加盐哈希存储防注入、防撞库(密码绝不明文存!)
定期备份数据库误删/被黑/硬盘坏都能恢复
密码存储:绝不明文

用户密码永远不能明文存进数据库。要用 bcrypt / argon2 这类专门的算法做"加盐哈希"——即使数据库被脱库,攻击者也拿不到原始密码。这是登录功能的底线。

4.9 进阶:Docker、CI/CD、监控(知道有这回事)

第一版网站不需要这些,但它们是你"未来要掌握"的方向,先建立概念:

  • Docker(容器):把"程序 + 它依赖的整个环境"打包成一个标准化的"集装箱",在任何机器上一条命令跑起来,彻底解决"我电脑上能跑、服务器上不行"。强烈建议网站稳定后学一下,部署体验质变。
  • CI/CD(持续集成/部署):用 GitHub Actions 等,实现"git push 一推,自动测试 + 自动部署到服务器",告别手动上传。
  • 监控与日志:网站挂了要第一时间知道。了解 journalctl 看 systemd 日志、Nginx 的 access.log/error.log,以后可上 Prometheus + Grafana、或 UptimeRobot 这类在线监测。
  • 负载均衡 / CDN:访问量大了,用 CDN(如 Cloudflare)把静态资源缓存到全球节点加速,用负载均衡把流量分到多台服务器。
第 4 部分小结:动态网站的完整链路

前端(HTML/CSS/JS)→ fetch 调 → 后端 API(Flask/FastAPI)→ 读写数据库。部署时:代码上 VPS → gunicorn 跑后端 → Nginx 反向代理 → certbot 上 HTTPS → systemd 守护 → 防火墙 + 加固。把这条链路亲手走通一次,你对"网站"的理解会彻底从模糊变清晰。

PART 5项目二:个人代理(梯子)的原理与搭建

先说清楚定位与边界

这一部分把代理当作"理解加密隧道如何工作"的网络工程实验,正当用途是访问 GitHub、Google Scholar、arXiv、Hugging Face、Stack Overflow、npm/pip/conda 等开发与学术资源。请仅用于个人合法用途,遵守你所在地的法律法规与服务商条款,不用于任何违法或商业牟利目的。技术原理本身是公开的网络知识,理解它对你这个 AI 专业的学生有长期价值。

5.1 原理总览:代理到底在做什么

正常上网是"你 → 直接访问目标网站"。代理是在中间插一个替身:你先把请求加密发给自己在海外的服务器,由它替你去访问目标,再把结果加密传回。对目标网站来说,访客是你的服务器;对你和服务器之间的链路来说,传的是看不懂的加密流量。

术语:代理 vs VPN vs 隧道

代理(Proxy):按应用/按流量转发(如浏览器走代理)。VPN:在系统层建一条虚拟网卡,把整机流量都套进加密隧道。隧道(Tunnel):泛指"把一种流量包进另一条加密通道"的做法。个人自用场景里,大家说的"梯子"多是加密代理,技术上和 VPN 高度重叠。

5.2 代理的本质:就是"转发"

剥到最底层,代理服务器干的事极其朴素——流量转发。最经典的协议是 SOCKS5:你的程序把"我要连 github.com:443"这个意图告诉代理,代理替你建立连接、来回搬运字节。它本身不加密。

所以一个"裸 SOCKS5/HTTP 代理"在公网上有两个问题:① 明文,链路上能被看到你在访问什么;② 容易被识别和阻断。解决这两点,就是后面"加密 + 伪装"的全部动机。

动手感受转发(本地玩)

很多工具能一行起一个本地 SOCKS5 代理。最容易体会的是 SSH 自带的功能:ssh -D 1080 root@你的VPS 会在本机 1080 端口开一个 SOCKS5 代理,流量经 SSH 加密隧道从 VPS 出去。把浏览器代理设成 127.0.0.1:1080 就能用——这其实已经是一个"最简陋的梯子"了,正好印证原理。只是它的流量特征明显、速度一般,所以才有了后面更专业的方案。

5.3 为什么"裸代理"会被识别:DPI 与流量特征

要理解现代代理协议为什么长那样,得先理解它们在对抗什么。网络中间设备(运营商、网关、防火墙)可以做 DPI(深度包检测)——不只看包要去哪,还分析包的内容和特征。识别一个连接是不是代理,常见三种手段:

  • 特征指纹:某些代理协议握手时有固定字节模式,一眼能认出来。
  • 流量行为:连接时长、包大小分布、时序等统计特征异常。
  • 主动探测:对可疑服务器主动发起连接试探,看它对乱七八糟的请求如何反应——真网站和代理服务器的反应不一样。
这条主线串起整个协议演进

代理技术这十年的进化,核心就一句话:让加密流量越来越像"一次再普通不过的 HTTPS 访问",从而既看不出内容(加密),也看不出它是代理(伪装),还经得起主动探测。下一节就是这条进化线。

5.4 加密与伪装:协议演进这条线

按出现顺序理解,你就明白每一代在补上一代的什么短板:

协议核心思路补了什么短板
Shadowsocks (SS)轻量加密代理,把流量加密成"看不出内容"解决明文问题;曾经够用,后被特征识别
V2Ray / VMess可扩展的传输框架,支持多种伪装(走 WebSocket、套 CDN 等)增强伪装与灵活性
Trojan干脆伪装成标准 HTTPS,跑在 443,配真实域名+证书;探测时回落到一个真网站对抗主动探测,"长得就像正经网站"
VLESS + XTLS更轻量(去掉冗余加密),性能更高减少开销、提速
VLESS + Vision + REALITY(Xray)直接借用一个真实大网站的 TLS 握手,无需自己买域名和证书,探测者看到的就是那个真网站当前主流之一,抗探测强、部署省事
Hysteria2 / TUIC基于 QUIC(UDP),在丢包高的线路上更快解决高延迟/丢包线路体验
术语:Xray-core 与 sing-box

你不用自己实现这些协议,有成熟内核:
· Xray-core(XTLS 项目):VLESS / Vision / Reality 的主力实现。
· sing-box:一个"全家桶"内核,服务端客户端通吃,支持上面几乎所有协议,配置统一,近年很流行。
新手 2026 年起步,推荐二选一:Xray 的 VLESS+Reality(省去域名证书),或 sing-box(一套搞定多协议)。

5.5 实操:在海外 VPS 上搭一套(以 Xray + Reality 为例)

命令与配置以官方仓库为准

代理内核更新较快,下面给的是流程骨架和代表性配置,具体安装命令、配置字段请对照官方最新文档:Xray-install(github.com/XTLS/Xray-install)、Xray-core(github.com/XTLS/Xray-core)、sing-box(github.com/SagerNet/sing-box)。理解每一步在干什么,比照抄命令更重要。

前提:已买好一台海外 VPS,已用 SSH 登录(第 3 部分),已做基础加固。

# ① 用官方脚本安装 Xray(命令以官方 README 为准)
bash -c "$(curl -L https://github.com/XTLS/Xray-install/raw/main/install-release.sh)" @ install

# ② 生成所需的随机凭据
xray uuid                 # 生成一个 UUID,当作你的客户端 id
xray x25519               # 生成 Reality 的 私钥/公钥 一对

然后编辑 Xray 的配置文件(通常 /usr/local/etc/xray/config.json)。Reality 的核心思路:让你的服务器在被探测时,表现得就像在访问 dest 指定的那个真实大网站。代表性配置骨架:

{
  "inbounds": [{
    "port": 443,
    "protocol": "vless",
    "settings": {
      "clients": [{ "id": "上一步的 UUID", "flow": "xtls-rprx-vision" }],
      "decryption": "none"
    },
    "streamSettings": {
      "network": "tcp",
      "security": "reality",
      "realitySettings": {
        "dest": "www.microsoft.com:443",       // 借用的真实网站
        "serverNames": ["www.microsoft.com"],   // 对外宣称的域名
        "privateKey": "x25519 生成的私钥",
        "shortIds": [""]
      }
    }
  }],
  "outbounds": [{ "protocol": "freedom" }]
}
# ③ 重启并设为开机自启,查看状态
sudo systemctl restart xray
sudo systemctl enable xray
sudo systemctl status xray      # 不正常就看日志:journalctl -u xray -e

# ④ 放行 443 端口(防火墙 + 云安全组都要)
sudo ufw allow 443

备选更省心的路线:用 sing-box,或借助社区成熟的一键脚本(认准高星、活跃维护的开源项目,谨慎对待来历不明的脚本)。无论哪种,理解上面这套"UUID + 密钥 + 借真实网站做 TLS"的逻辑都通用。

5.6 两条伪装路线:Reality 与"域名+证书"

让流量像正经 HTTPS,主要两条路,理解差异你就能自己选:

Reality(借别人网站)域名 + 证书(用自己网站)
要不要域名不需要需要,还要解析到 VPS
要不要证书不需要(借用目标站的)需要(Let's Encrypt,同 4.7)
对外长相像在访问那个被借的大网站像在访问你自己的网站
适合怕麻烦、想抗探测,首选想顺便真挂个网站当幌子(Trojan/VLESS+TLS)
两个项目在这里交汇

"域名+证书"这条路用到的域名解析(4.5)、Nginx、Let's Encrypt 证书(4.7),和你做网站时完全是同一套技能。甚至可以在同一台机器上,443 端口前面放 Nginx,把代理流量和真实网站流量按规则分流——这就是为什么这份教程把两个项目放一起讲:底层知识是共享的。

5.7 客户端与日常维护

服务器端搭好后,在你的设备上用客户端填入服务器地址、端口、UUID、公钥等参数即可连。常见客户端:

平台常见客户端
Windowsv2rayN、sing-box、Nekoray、Clash 系
macOSsing-box、V2rayU、Clash 系
Androidv2rayNG、sing-box、Clash Meta 系
iOSShadowrocket、sing-box(均需对应区账号)
术语:分流规则

客户端通常支持分流:国内网站直连(更快),只有需要的流量才走代理。这套规则一般写在配置或订阅里(如 Clash 的规则、GeoIP/GeoSite 数据)。理解了 1.3 的 IP、1.5 的域名,你就知道分流就是"按目标 IP/域名决定走代理还是直连"。

维护要点

  • 安全:VPS 同样要做第 3、4 部分的加固(密钥登录、防火墙、更新)。代理服务器被入侵,后果不小。
  • 稳定:关注内核版本更新;journalctl -u xray -e 看日志排错。
  • 合规自律:控制在个人合法用途内,不分享牟利、不做违法访问。把它当学习工具善用。
  • 备份配置:把 UUID、密钥、配置文件记好,换服务器能快速重建。
第 5 部分小结
  • 代理本质是替身转发;裸代理明文且易被识别。
  • 对抗 DPI / 主动探测,协议一路演进:SS → V2Ray → Trojan → VLESS+Reality → Hysteria2,主线是"越来越像普通 HTTPS"。
  • 实操:海外 VPS + Xray(VLESS+Reality) 或 sing-box,服务端配好 → 客户端连入 → 分流使用。
  • 所用的域名/证书/Nginx 技能与网站项目完全共享。仅限个人合法用途。

C. 排错思路:从下往上,逐层定位

网络问题最忌乱猜。按分层模型从底往上排,几步就能锁定:

  1. 通不通? ping IP。不通 → 机器没开、IP 错、或网络层不通。
  2. 域名解析对吗? nslookup 域名。解析到错误 IP → 改 DNS 记录、等缓存。
  3. 端口开着吗? 服务没起 / 防火墙没放行,是 80% 的"连不上"。查 systemctl status + ufw status + 云安全组。
  4. 服务在跑吗? systemctl status xxx、journalctl -u xxx -e 看它有没有崩、报什么错。
  5. 应用层报错? 看状态码:502 后端没起、404 路径错、500 代码错、证书错则查 HTTPS 配置。
  6. 逐段二分:本机能连吗(curl 127.0.0.1:端口)?服务器本地能连吗?换个网络呢?把链路切成几段,定位是哪一段断的。
一句话心法

"先确认底层通,再往上层查;每次只改一个变量,改完就验证。" 这套思路对网站、代理、以及未来你遇到的任何网络问题,都通用。
